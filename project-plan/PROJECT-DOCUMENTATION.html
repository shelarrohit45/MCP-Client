1. Project Goal

Build a local DevOps automation client that monitors a GitHub repository and sends email notifications — with an optional natural language AI agent that uses the same workflows.

Primary objectives

  1. Connect to GitHub and Email through the Model Context Protocol (MCP) — not custom API glue
  2. Automate CI failure alerts, daily digests, and PR notifications
  3. Run entirely locally — no AWS or custom cloud hosting for core automation
  4. Add an optional LLM agent (ask) with Firebase Firestore for chat memory and audit history
  5. Follow industry-standard patterns: secrets in env, dry-run modes, dedup, structured logging, human confirmation for sensitive actions
Success criteria: Scheduler runs digest + CI checks without the agent. Agent layer is additive — OpenRouter + Firebase are only required for ask, llm-test, firebase-test, and agent-history.

2. Project Overview

MCP DevOps Client is a Python application that connects to two MCP servers — GitHub MCP and Email MCP — to automate repository monitoring and email notifications. Step 11 adds OpenRouter (LLM) and Firebase Firestore (agent memory).

What the project does

  • Fetches GitHub data: open PRs, CI failures/successes, open issues, latest release
  • Sends CI failure alert emails when checks fail (last 24 hours)
  • Sends a daily digest email summarizing repo activity
  • Sends PR event emails: created, pushed, merged, rejected, approved, CI passed/failed
  • Supports Merge/Reject buttons in PR emails via a local HTTP action server
  • Prevents duplicate CI alerts using logs/state.json
  • Runs digest and CI checks on a schedule via APScheduler
  • Natural language ask command with tool calling (fetch data, preview/send digest, CI alerts)
  • Stores chat sessions, LLM runs, and workflow audit in Firebase Firestore
  • Logs all activity to logs/app.log

Agent data flow (ask command)

  1. User runs python src/main.py ask "..."
  2. agent_chat.py loads prior messages from Firestore (agent_sessions/{id}/messages)
  3. agent_loop.py sends conversation + tool schemas to OpenRouter
  4. Model may return tool calls → agent_tools.py runs wrapped workflows (e.g. fetch_github_summary)
  5. Sensitive tools pause for confirmation unless --yes
  6. Final answer saved to Firestore; run metadata logged to agent_runs

9. Firebase Firestore — Implementation & Testing

Why Firestore (not SQLite or local files)?

  • Durable chat memory — sessions survive CLI restarts and machine reboots
  • Audit trail — every LLM run and tool workflow logged for observability
  • Queryable history — list recent sessions, runs, workflow entries
  • Industry pattern — managed cloud DB with service account auth for backend agents
  • Local logs stay local — logs/app.log for debugging; Firestore for durable audit

How we implemented it (src/firebase_store.py)

  1. Initialize once — firebase_admin with service account certificate per process
  2. Write APIs — save_message(), log_agent_run(), log_workflow_history()
  3. Read APIs — get_session_messages(), list_recent_sessions(), list_recent_agent_runs()
  4. Connectivity test — run_firebase_connectivity_test() for firebase-test CLI
  5. Observability fields — latency_ms, llm_calls, tools_called, token counts on each run

Firestore collections schema

CollectionDocument pathKey fields
agent_sessions agent_sessions/{sessionId} title, created_at, updated_at
messages (subcollection) agent_sessions/{id}/messages/{msgId} role (user/assistant/tool), content, tool_name
agent_runs agent_runs/{runId} session_id, model, prompt_tokens, completion_tokens, tools_called, latency_ms, success
workflow_history workflow_history/{entryId} workflow, status, summary, metadata.session_id

15. Workflows Explained

CI Alert

  1. Search GitHub for PRs with status:failure in last 24h
  2. Skip failures already in logs/state.json
  3. Render templates/ci_alert.html and send via Email MCP

Daily Digest

  1. Fetch open PRs, issues, CI pass/fail, latest release
  2. Render templates/digest.html
  3. Send with subject [Daily Digest] owner/repo — date

Agent ask

  1. Load Firestore session history
  2. LLM decides whether to call tools
  3. Tools execute workflows (with guardrails for sensitive ones)
  4. Response + run metadata saved to Firestore
